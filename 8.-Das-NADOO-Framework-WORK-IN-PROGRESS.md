Das Nadoo Framework ist eine Art und weise wie Projekte bzw. Code strukturiert wird.


Was ist ein Framework?

Eine Struktur für das Entwickeln von Applikationen. Diese Struktur ermöglicht es leichter Code mit anderen Projekten die dieser Struktur folgen auszutauschen. Es ermöglicht Entwicklern auch in andere Projekte die dieser Struktur folgen leichter beizutreten bzw. diese zu verstehen oder sich darin zu orientieren.

Der größte Teil der Zeit die ein Softwareentwickler mit Arbeiten verbringt ist nicht das verfassen von Code sondern das lesen und verstehen des Codes den Er/Sie bzw. ein anderer Entwickler geschrieben hat. Ein großer Teil davon ist das verstehen der Struktur. Dadruch das Code an sich kein bzw. nur minimale Vorgaben macht wie der Code zu struktuieren ist steht es jedem Entwickler frei seinen Code so zu strukturieren wie es Ihm/Ihr grade recht ist. Dadruch haben sie viele "Syles" Code zu schreiben entwicklet. Diese sind nochmal zu trennen von den Modellen wie MVC, MVP, MVVM.

Besonders erfahrenere Entwickler greifen häufig auf Frameworks zurück da diese eine einheitlichere Struktur versprechen und so es leichter machen auf bestehende Projekte zurückzugreifen bzw. kommen diese bereits mit viele Funktionen die einem dabei helfen schneller seine Programmierziele umzusetzen bzw. wie zuvor erwähnt wenn später andere Entwickler das Projekt übernehmen oder beitreten sich darin zu orienteiren.

Einige Beispiele für solche Frameworks:

Node.js, Vue, React, Angular (JavaScript)
SpringBoot(Java)
Django, Flask, FastAPI (Python)
Wordpress, Laravel, Sinphony (PHP)
...

Ein weiterer Vorteil solcher Frameworks ist es das Sie viele der Anfängerfehler verhindern können bzw. basieren sind auf den Erfahrungen von Entwicklern die bereits Herrausfoderungen gelöst haben.

Tatsächlich wenn Sie diesen Punkt im Wiki erreicht haben, haben sie bereits mit zwei solcher Frameworks gearbeitet.

Briefcase und Toga.

Briefcase ist ein Framework zum strukturieren von Projekten das durch diese Struktur es Ihnen ermöglicht Ihre Software für MacOS, Windows und viele andere Plattformen zu packen und so einfach zu verteilen.

Toga ist ein Framework zur gestaltung von Interfaces die so auf den Plattformen wie native Applikationen aussehen.

Hierbei lassen aber beide und auch viele der anderen genannten Frameworks den Enwicklern noch sehr viele Freiheiten. Diese Freiheiten haben Ihre vor und nachteile. Besonders für erfahrene Entwickler kann sich das einschränken der Freiheiten wie Sie Ihren Code gestallten unangenehm sein da Sie nicht unbedingt auf bereits erlangte Gewohnheiten zurückgreifen können und sich stattdessen an das Framework anpassen müssen um von deren positiven Effekten zu profitieren. Auch gibt es situationen in den Frameworks zu "mächtig" sind. Also schon mit zu vielen "Batteries" kommen die an diesem Punkt noch nicht notwenig sind.

Ein Beispiel ist hier Django und Flask.

Beides sind Frameworks zur gestaltung von Webanwendungen. Django hierbei kommt bereits mit eine recht fixen Struktur, Usermanagment, Datenbank, ORM und und und. Flast dagegen beinhaltet praktisch nichts davon. Jedoch ist es häufig so das eine Idee vielleicht klein anfängt und schlussendlich auch eine Flask App all die Bausteine die in Django vorhanden sind benötigt. Hierbei wird dann auf externe Plugins bzw. Packages gesetzt während Django hier tiefintegrierung vorzieht um dem Nutzer eine bessere Entwicklungserfahrung zu bieten.


Das NADOO Framework ist ähnlich wie Django. Es ist ein Framework mit "Batteries included". Es ist ein zusammenschluss von Briefcase, Toga, zmq(Messeging und Networking) und verbindet dies mit klaren Codingconventionen.

Hierbei bietet das NADOO Framework diese Struktur nur an. Der Grund für die Stuktur ist das solang sich der Entwickler an die Struktur hält er durch die Werkzeuge in NADOO Launchpad beim entwickeln unterstütz wird.

Das NADOO Framework ist von grund auf darauf entwickelt die schwiegigsten Probleme der App Entwicklung leicht zu machen. Durch das Bennenungsschema wird die schwirgeste aller Entwicklungsaufgaben weitgehend gelöst. Die Frage nach der Bennenung. Richtig gelesen. Bennenung. Zwar setzt NADOO auch auf eine so einzigartige Struktur aber im Kern ist das wichtigste am Framwork die Regeln zur benneung von Funktionen, Klassen und anderen Kompnenten. Das soll auf lange Zeit sowohl den Enwicklern es einfach machen schnell in ein Projekt einzusteigen, neue Feature zu entwickeln und die Zusammenarbeit mit KI zu vereinfachne.

Eins der Kernprobleme bei der Zusammenarbeit mit KI ist das KI schnell "verwirrt" wird sobald die Menge des Codes groß wird. Besonders zu anfang neigen Entwickler dazu riesige Scripte zu erstellen. Wenn die KI dann bei der Entwicklung helfen soll muss diese die kompletten Scripte "überblicken" und neigt dazu sich sowhl zu wiederholen, also Funktionen die bereits existieren neu zu definieren, Logic an mehrern stellen neu zu definiern bzw. auch code zu schreiben der keine Funktion hat. Um dies einzuschränken haben wir klare Regeln. KI ist im Kern nicht inteligent. Nur weil wir Sie künstliche Inteligenz nennen bedeutet dies noch lange nicht das Sie intelligent ist. Tatsächlich funktionieren diese Systeme rein darauf das nächste Wort vorherzusagen. Nicht viel anders wie die Wortvorschläge der Handytastatur. Hierbei werden Wahrscheinlichkeitsrechnungen  druchgeführt bei denen dann das wahrscheinlichste Wort bzw. Token dem Text hinzugefügt wird. Hierbei greift die KI auf riesige Mengen an Daten zu. Das kernproblem dabei ist das ohne klare vorgaben versucht die KI zu "erahnen" was die richtige Antwort ist. Es wird versuchen anhand der Informationen die Ihr zur verfügung steht Antworten zu generieren. Sind aber die Informationen nicht nur zu viel sondern auch mit viel Varianz versehen wird die produktion der Antwort ungenau und fehleranfällig. Ein weg also um die Genauigkeit zu erhöhen ist es der KI in form eines Promptes genau zu erklären wie der Code auszusehen hat den es generieren soll. Hier kommt das NADOO Framework wieder ins spiel. In unserem Framework ist klar definiert wie der Code zu definieren ist. Also wo wird die Scriptdatei platziert. Wie wird die Funktion/Klasse bennant. Wie stehen diese in verbindung miteinander. Wie sind die Abläufe bzw. funktionen zu füllen. Wie werden Interfaces definiert und nach vieles mehr. Hierbei streben wir eine sowohl für den Menschen und Maschine gut vorhersebaren Leitfaden vor. Das ist notwenid da die Aufgabe eines Entwicklers sich sowohl jetzt und den nächsten Jahren immer mehr dahin entwickelt dann einzugreifen wenn die KI nicht mehr "weiter weis". Dabei ist aber wichtig das der Code der so entsteht auch weiterhin für einen Menschen gut lesbar ist bzw. dies sogar noch mehr ist. Leider schreiben jetzt schon viele Entwickler um zeit zu sparen Code in stark abgekürzter form und machen es so Lesern später schwerer ohne verständnis über kompliziertere Konzepte der Sprache diese lesen zu können. Unser Ziel mit dem NADOO Framework ist voreilige optimierungen zu vermeiden bzw. sogar zu verhindern und dann wenn notwenig umzusetzten.

Nachdem wir jetzt etwas über die Hintergründe aufgelistet haben, hier was die Bateries des NADOO Framework sind:

Briefacse -> Erstellen von nativen Installionsdatei für MacOS, Windows, Linux, Android, iOs, Web, Terminal
Toga -> Erstellen von nativen Intefaces für s.o.
ZMQ -> Ermöglicht es zwischen verschiendenen Applikationen zu kommunizieren, Kommunikation im Netzwerk oder auch über das Internet, Ermöglicht es auf alle von ZMQ unterstützten Programmiersprachen Ökosysteme zuzugreigen(liste der Programmiersprachen)
NADOO Launchpad -> Überwacht, Manged und unterstützt beim Entwicklungsprozess und übernimmt viele der Aufgaben die bei der Entwicklung anfallen

Die Konzepte des Framworks zur Strukturierung des Codes sind simpel.

Regel 1.

Eine Funktion = Eine Datei

Während wir es vor dem NADOO Framework gewohnt waren mehrere Funktionen in Script Datein (.py) zu vereinen wird im NADOO Framework strikt eine Datei eine Funktion gelebt. Dabei trägt die Datei den Namen der Funktion.

Das bringt einige Vorteile mit sich.

1. Abhängigkeiten sind sofort durch die Imports und Parameter klar erkennbar.
2. Keine Disorientierung innerhalb des Scriptes, weil man zwischen funktionen die in einem mehrere tausend Zeilen langen Script verteilt sind hin und her springt
3. Keine Frage zur bennenung der Script Datei. Scriptname = Dateiname
4. Im Dateibaum sind sofort die funktionen sichtbar und nicht innerhalb der Datei versteckt
5. über cmd+p bzw. windows strg+p kann der name eingegeben werden und man springt sofort in die richtige Datei ohne die notwenigkeit von Suchen
6. Importe von Funktionen werden erleichtert da die Struktur immer identisch ist. Dies führt zu importen wie "from .get_app_name import *"
7. KI braucht nicht unnötigen Kontext mitbeachten
8. Keine Funktions duplikation da namen nicht doppelt vergeben werden können!
9. Einfache teilbarkeit von einzelner funktionalität da Funktionen nicht erst aus anderen Struturen extrahiert werden brauchen. Stattdessen kann die Funktionsdatei und die davon abhänigen kopiert und in ein anderes Projekt koppiert werden.

Das sind nur einige der Vorteile dieses einen Punktes des Frameworks.

Leider kommt dies bzw. würde dies auch mit einem klaren Nachteil kommen der gegen das geht wofür Python so beliebt ist.

"Biolerplate"

Boilerplate ist ein Fachwort für nicht funktionalen Code, der nur dafür notwenig ist damit man überhaupt funktionalen Code schreiben kann. Einer der großen Vorteile den Python gegenüber anderen Programmiersprachen war es bzw. ist es immer noch wie wenig notwenig ist um Funktionscode zu schreiben. Hier reicht wie sie bereits erlebt haben normalerweise einfach das erstellen einer .py Datei aus und man kann anfangen Code zu schreiben. Diese Datei kann dann druch python dateiname.py einfach ausgeführt werden.

Im NADOO-Framework jedoch müssen wir nun für jede Funktion eine eigene Datei erstellen, funktionen die wir nutzten wollen importieren und sind angehalten uns an fixe Namens und Ordernstrukturen zu halten. Dies erscheint im ersten Augenblick zusätzlicher aufwand und wieder etwas neues an das gedacht werden muss. 

Dem ist aber nicht so! 

Zumindest nicht wenn mit NADOO-Launchpad gearbeitet wird. 

NADOO-Launchpad ist unser zentrales Entwicklungsunterstützungsprogram. So wie Sie bereits briefcase dev, briefcase new usw. kennengelernt haben und noch viele andere Werkzeuge mit denen Sie Aufgaben der Verwaltung Ihrer Projekte erledigen ist NADOO-Launchpad ein weiteres. Lauchpad dagen ist um die Benutzerfreundlichkeit zu verbessern vorerst ein GUI Tool. Es vereint viele der von Ihnen bereits gelernten Werkzeuge bzw. wird dies. Was aber NADOO-Lauchpad bereits jetzt kann ist sie dabei zu unterstützen sich an die Regeln des Frameworks zu halten. So werden Funktionen die sie in Ihrem Code verwenden automatisch importiert, sollten Sie eine neue Funktion definieren bzw. verwenden diese in Ihrem Code so wird eine neue Datei für diese Funktion automatisch angelegt und entsprechend des Funktionsnamens angepasst. Auch können Sie den Namen der Funktion jederzeit direkt im Code anpassen und alle verwendungen werden angepasst und die Datei mit dem neuen Namen neu erstellt(Achten Sie hierbei darauf, dass Sie sich nach dem Umbennenen eventuell noch in der alten Datei befinden).

Zu NADOO-Launchpad später mehr.


Ordnerstruktur:

Da sie breits Ihre ersten Apps mit briefcase und Toga erstellt haben sollten Sie bemerkt haben das briefcase eine bestimmte Ordnerstruktur vorgibt bzw. wenn Sie mit briefcase new ein neues Projekt erstellen Sie eine Stuktur vorfinden.

Das NADOO-Framework baut auf dieser Struktur auf eine etwas ungewöhnliche art auf. Was Ihnen im verlauf Ihrer Karriere beim arbeiten mit anderen Frameworks auffallen wird ist das eine der größten Herrausforderungen das Organsieren Ihres Codes sein wird. Über die Jahre haben sich hier viele verschiedene Lösungsansätze entwickelt. Hierbei kommt es oft dazu, dass sich sogar funktionalität hinter der Struktur des Codes, also in welchen Ordnern sich datein befinden funktionalität versteckt. So gibt es Webframeworks(**gib hier beispiel**) bei denen es Ordner gibt die sogenante Route verstecken. Dies sind die Adressen die sie bei z.B. www.google.com/das_hier_ist_eine_route 
Dies hat wie alles seine Vor und Nachteile. Ein klarer Nachteil ist das dies nicht sofort für jemand der noch nicht mit dem Framework vertraut ist ersichtlich ist. Auf der anderen Seite bitete es eine "Abkürzung" für Entwickler da Sie weniger damit beschäftig sind Ihre App zu Konfigurieren und sich auf das erstellen der App konzentrieren können. Jedoch wie bereits erwähnt basieren solche Frameworks nach meiner Einschätzung auf einer klaren Fehlannahme. Sie bevorzugen vermeintliche Entwicklungsgeschwindigkeitsgewinne und vernachlässigen dafür lesbarkeit bzw. erfodern eine steile Lernkurve.

Auch werden häufig Ordner und Datein dafür verwendet Code zu "Organisieren". Code zu ordnen ist eine gewaltige Herrausforderung. Im Kern können wir sogar soweit gehen zu sagen, dass es diese Herrausforderung war die zur Spaltung der Softwareentwicklung führte. So etablierten sich zwei mehr oder weniger getrennte Arten Code zu Strukturieren. In Ihrer Ausbildung bzw. über Videos haben Sie vielleicht schon von Funktionaler, Imperativer, Prozedualer und Objektorienteirter Programmierung gehört. Dies sind größtenteils nicht viel mehr als verschiedene Arten von Organisation von Code. Wie verwaltet man Funktionaltiät und Daten. Werden diese Gebündelt in Klassen oder liegen diese getrennt von einander vor. In den wenigsten Fällen wird rein Funktional oder rein Objektorientiert gearbeitet. Wie so häufig im Leben gilt "Das richtige Werkzeug für die richtige Situation" Zu erkennen wo und wie man diese Anwendet ist was Sie in den nächsten Jahre lernen werden. Das NADOO Framework soll Ihnen dabei helfen
